# CSV Reports

> Coming soon to [v3.77.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v3.77.0)

Gwen can also generate CSV reports for you.

## Configure CSV files

By default, Gwen will generate the following CSV files under `reports/results` subdirectory in your configured [`output`](/docs/settings#gwen-outDir) directory, but you can completely reconfigure this to suit your own requirements:
- Feature level results:
  - feature-results-PASSED.csv - A log of all passed feature results
  - feature-results-FAILED.csv - A log of all failed feature results including the failed reason
  - feature-results-ALL.csv - A log of all feature results (passed or failed) including the failed reason (for failed)
- Scenario level results:
  - scenario-results-PASSED.csv - A log of all passed scenario results
  - scenario-results-FAILED.csv - A log of all failed scenario results including the failed reason
  - scenario-results-ALL.csv - A log of all scenario results (passed or failed) including the failed reason (for failed)

### Configuration

Files can be configured within the `gwen.reports.results.files` element in your settings file.
```json
<name> {
  file = "<filename>" 
  scope = "[scope]"
  status = "[status]"
  fields = [
    { name = "<fieldName 1>", ref = [reference 1], optional = [optionality 1] }
    { name = "<fieldName 2>", ref = [reference 2], optional = [optionality 2] }
    ..
    { name = "<fieldName N>", ref = [reference N], optional = [optionality N] }
  ]
}
```

Where:

<ul>
  <li><code>&lt;name&gt;</code> - (Mandatory) Is an arbitatry name for the file element</li>
  <li>
    <code>&lt;filename&gt;</code> - (Mandatory) - Is the name or path to the target CSV file
    <ul>
      <li>If a file name only is specified (without a prefixed path), then the named file will be written to the `reports/results` subdirectory within your configured [`output`](/docs/settings#gwen-outDir) directory</li>
      <li>If the file name is prefixed with a path, then the named file will be written to that location</li>
    </ul>
  </li>
  <li>
    <code>[scope]</code> - (Optional) is either:
    <ul>
      <li><code>Feature</code> - To log the results of all features to the CSV file as they complete</li>
      <li><code>Feature: &lt;name&gt;</code> - To log the results of a specifically named feature only to the CSV file as it completes, where <code>&lt;name&gt;</code> is the name of the feature to log the results for</li>
      <li><code>Scenario</code> - To log the results of all scenarios to the CSV file as they complete</li>
      <li><code>Scenario: &lt;name&gt;</code> - To log the results of a specifically named scenario only to the CSV file as it completes, where <code>&lt;name&gt;</code> is the name of the scenario to log the results for</li>
      <li><code>StepDef</code> - To log the results of all <Link to="/docs/meta#stepdefs">StepDefs</Link> to the CSV file as they complete</li>
      <li><code>StepDef: &lt;name&gt;</code> - To log the results of a specifically named StepDef only to the CSV file as it completes, where <code>&lt;name&gt;</code> is the name of the StepDef to log the results for</li>
      <li>Or blank to not log any results automatically (for when you want explicitly manage CSV logging yourself)</li>
    </ul>
  </li>
  <li>
    <code>[status]</code> - (Optional) is either:
    <ul>
      <li><code>Passed</code> - To log only passed results to the CSV file</li>
      <li><code>Failed</code> - To log only failed results to the CSV file</li>
      <li>Or blank to log passed or failed results to the CSV file</li>
    </ul>
  </li>
  <li><code>fields</code> - (Mandatory) - Contains an array of fields as described below</li>
</ul>

Fields are configured as arrays witihin the `fields` attribute and have the following structure:

```json
{ name = "<fieldName>", ref = [reference], optional = [optionality] }
```

Where:
<ul>
  <li><code>&lt;fieldName&gt;</code> - (Mandatory) - Is the name of the field as it will appear in generated CSV</li>
  <li>
    <code>[reference]</code> - (Optional) - Can be one of:
    <ul>
      <li>The name of a varaible in scope containing the value to bind</li>
      <li>If blank, will use the value bound to the variable of same name as <code>&lt;fieldName&gt;</code></li>
      <li>
        If <code>&lt;fieldName&gt;</code> is set to <code>*</code>, then:
        <ul>
          <li>This reference must specify a path to the CSV template file containing the names of all fields to bind in the first row (and the names of such fields will also be used as the binding references)</li>
          <li>Can also be set to the special reference <code>$&lt;gwen.options.dataFile&gt;</code> which will resolve to the input CSV (data feed) file passed to Gwen via the [<code>-i|--input-data &lt;file&gt;</code>](/docs/cli#gwen-cli-options-input-data) CLI option. In the case all the fields present in the input file will be included and resolved by name.</li>
        </ul>
      </li>
    </ul>
  </li>
  <li>
    <code>[optionality]</code> - (Optional) - Can be either of:
    <ul>
      <li><code>true</code> - (Default) - To make this field mandatory. Will result in an undefined binding error being raised at runtime if the field binding does does not exist (i.e: is not in scope) at the time the record is written. </li>
      <li><code>false</code> - To make this field optional. Will resolve to blank at runtime if the field binding does does not exist (i.e: is not in scope) at the time the record is written. </li>
    </ul>
  </li>
</ul>

:::note Example

Executing features in the `gwen/features/todo` folder and log CSV reports.

yarn  -b f html,results gwen/features/todo
