Get Started

Before you start

Read the quick Gwen introduction.

System requirements#

  • Linux, Mac or Windows
  • Chrome, Firefox, Safari, or Edge or IE web browser
  • Java JRE 8+ (version 1.8 or higher)
    • You can check if you have Java installed by opening a command prompt and typing java -version. A version will be displayed if you do have it installed (install Java otherwise).
  • node
    • Required if you will be using Gwen in a JS project with npm (install node).

Install Gwen#

  • Add to JS Project
  • Install Manually
  • Install Workspace (Discontinued!)

Adding Gwen to a JS project is the recommended approach. If you don't have one, you can create one by installing node and running npm init -y in a new directory and use that.

Run the following command in the root of your project to add Gwen to it as a dev dependency:

  • npm
  • Yarn
npm i --save-dev @gweninterpreter/gwen-web
yarn add -D @gweninterpreter/gwen-web
Automatic Updates

Automatic installs and updates to the latest version of Gwen-Web are enabled by default for JS projects.

You can disable automatic updates by setting the version field in the gwenWeb section of your package.json file to a specific version.

"gwenWeb": {  "version": "3.0.0"}

Setting it to latest will re-enable automatic updates.

Perform the following to manually install Gwen on your machine and add it to your system path:

  1. Browse the Gwen-Web releases and download the binary ZIP package of the version you want to install
  2. Extract the downloaded ZIP into a desired location on your drive
  3. Add that location to your system PATH variable
No Automatic Updates

There are no automatic updates with the manual approach. You will need to repeat this process when you want to upgrade or downgrade your installed Gwen verison. You will also need to do the same on your build server(s) if you plan to run Gwen there.

Discontinued

Gwen workspaces have been discontinued as of Gwen 3 in favor of JS projects. To migrate:

  1. Install node and run npm init -y in a new directory to make it a JS project
  2. Add Gwen to it and initialise it as described in this section and the next
  3. Copy your feature and meta files across
  4. Tweak settings to match those you had in your workspace

Initialise your project directory#

  • JS Project
  • Standalone Project

Run the following command in the root of your project where you added Gwen as a dev dependency:

  • npm
  • Yarn
npm run gwen init
yarn gwen init

Set your default browser#

Gwen is configured to use the Chrome browser by default. Perform the following to change that to another browser if you wish.

  • JS Project
  • Standalone Project

Amend the default gwen.cli.options.config setting in the gwen.conf file in your project to use the chrome, firefox, safari, edge, or ie configuration.

gwen {  ..  cli {    options {      ..      config = [        "gwen/browsers/chrome.conf", <<-- Amend this to your preferred browser conf        "gwen/env/local.conf"      ]      ..    }  }  ..}

Amend the default gwen.cli.options.config setting in the gwen.conf file in your project to use the chrome, firefox, safari, edge, or ie configuration.

gwen {  ..  cli {    options {      ..      config = [        "browsers/chrome.conf", <<-- Amend this to your preferred browser conf        "env/local.conf"      ]      ..    }  }  ..}

Run the samples#

  • JS Project
  • Standalone Project

Run the following command in the root of your project to execute the sample features.

  • npm
  • Yarn
npm run gwen -b gwen/samples
yarn gwen -b gwen/samples
caution

This option assumes that Gwen is in your system path (that is: you manually installed Gwen on your machine).

Run the following command in your Standalone Project to execute the sample features.

gwen -b samples
Batch or REPL

The -b (or --batch) switch tells Gwen to exit immediately after executing the samples. Omitting it will open the Gwen REPL console instead after the samples have executed.

See also the HTML evaluation reports generated in the output directory too.

Share with your team#

Once you're comfortable with all of the above, you might want to push your project (or Standalone Project) to a Git repository and share it with your team. A .gitignore file that excludes the output directory is included. You can keep or delete the samples subdirectory (up to you).
