Gwen Introduction

Automation Bindings for Gherkin#

Gwen is a Gherkin interpreter that executes feature specs to perform web automation through bindings in meta specs that you compose with the Gwen DSL.

How it works#

1. Declare your feature specs#

Declare feature specs to describe how scenarios and examples must behave. The following example describes how adding items to a todo list should behave.

Feature: Add todo items   Scenario: Create todo list    Given a new todo list     When the following items are added          | Item          |          | Get the milk  |          | Walk the dog  |     Then the list will contain 2 items

Feature specs are used to communicate requirements that are clear, concise and declarative. Gwen aims to take these requirements "as is" and make them executable to drive automated testing both during and after development. It interprets each step and evaluates it using an embedded automation engine that executes to perform operations and checks consistently and reliably across browsers and platforms according to bindings that you specify in meta.

2. Compose your meta specs#

Compose meta specs to describe how your features are to be evaluated. The following example describes how the above feature might be tested against a todo application.

Feature: Todo Meta

  @Context  @StepDef  Scenario: a new todo list      Given my todo list can be located by css ".todo-list"       When I navigate to "${todo.page.url}"       Then the page title should contain "TodoMVC"        And my todo list should not be displayed
  @Action  @StepDef  @DataTable(header="top")  @ForEach  Scenario: the following items are added      Given the todo field can be located by class "new-todo"        And my new item is "${data[Item]}"       When I enter my new item in the todo field       Then my todo list should contain my new item
  @Assertion  @StepDef  Scenario: the list will contain <expected count> items      Given the displayed count can be located by css ".todo-count strong"       Then the displayed count should be "$<expected count>"

Meta specs describe what Gwen will evaluate in Gherkin language and are necessarily imperative because they are used to define all the step definitions and bindings required for automated execution.

Declaring step definitions#

Each step definition is declared as a @StepDef annotated scenario with a name that matches that of the step in the feature it binds to. Each one calls out to one or more Gwen DSL steps (or other step definitions) which perform the specified operations at run time.

The first step definition in our meta example binds to the first step in the feature and will only execute when it is interpreted for execution by Gwen at runtime.

Feature Step
  Given a new todo list
Step Definition
  @Context  @StepDef  Scenario: a new todo list      Given my todo list can be located by css ".todo-list"       When I navigate to "${todo.page.url}"       Then the page title should contain "TodoMVC"        And my todo list should not be displayed

Web elements selectors#

The first thing the above will do is bind the selector for locating the todo list element on the web page to the name my todo list. Later steps can then reference the element by this name to interact with it and Gwen will locate it dynamically.

Step Definition 1, Step 1
  Given my todo list can be located by css ".todo-list"
Uses Gwen DSL Step:
  • <element> can be located by <id|name|tag name|tag|css selector|css|xpath|class name|class|link text|partial link text|javascript|js> "<expression>"

All the standard Selenium selectors are supported and additionally JavaScript selectors also, which are very handy for those special edge cases where the standard ones are not robust enough or where evaluating a function on a page may be necessary.

Navigating to a web page#

The next step in this step definition will open a new browser window and navigate to the URL in the todo.page.url property.

Step Definition 1, Step 2
  When I navigate to "${todo.page.url}"
Uses Gwen DSL Step:
  • I navigate to "<url>"

Accessing externalised properties#

We could have just hard coded the http://todomvc.com/examples/react URL in the above step, but externalising it enables us to target different environment specific URL's without changing our meta. This is useful during development when you want to execute against a locally running instance of the application or during testing when it is running on a different host or port. With this approach you can define a settings file per environment and assign the URL for that environment to the todo.page.url property accordingly in each. Then when you want to target an envionment, you just pass the settings file for that environment to Gwen and the URL will be resolved accordingly.

Checking the page title#

The next step in the same step definition checks that page title in the browser window contains the string literal "TodoMVC" in it. If it doesn't then an assertion error will be raised, otherwise execution will resume to the next step. It's generally a good idea when navigating to a page to verify that you did indeed land on that page before proceeding to interact with it. Checking the title is one way to do it and will suffice in this case.

Step Definition 1, Step 3
  Then the page title should contain "TodoMVC"
Uses Gwen DSL Step:
  • the page title <should|should not> <be|contain|start with|end with|match regex|match xpath|match template|match template file> "<expression>"

Verifying that an element is not displayed#

The last step in this step definition verifies that the web element containing the current list of todo items is not displayed. We expect it to not be because we haven't added any items to it yet. Recall that we bound this element in the verify first step earlier to the name my todo list. So here we reference it by that name and if Gwen cannot locate it then execution will resume, otherwhise an assertion error will be raised.

Step Definition 1, Steps 5
  And the displayed count should not be displayed
Uses Gwen DSL Step:
  • <element> <should|should not> be <displayed|hidden|checked|ticked|unchecked|unticked|enabled|disabled>

Processing records in data tables#

The second step definition specifies the @DataTable and @ForEach annotations which work together to process each record of data in the table of the second step in the feature.

Feature Step
  When the following items are added       | Item          |       | Get the milk  |       | Walk the dog  |
Step Definition
  @Action  @StepDef  @DataTable(header="top")  @ForEach  Scenario: the following items are added      Given the todo field can be located by class "new-todo"        And my new item is "${data[Item]}"       When I enter my new item in the todo field       Then my todo list should contain my new item

The body of the step definition will be evaluated for each record. Each field name in the table header will be bound to the respective value in each record as it is iterated over and these will be made available to the steps in the body of the step definition as data[name] where name is the name of the field.

In this example the sole field named Item is referenced using data[Item] in the second step of the body. This step binds the resolved value to a string named new todo item for use by the two steps that follow. We do this to eliminate the redundancy of having to resolve it twice and also to make maintenance easier should we ever decide to rename this field in the table.

Step Definition 2, Step 2
  And my new item is "${data[Item]}"
Uses Gwen DSL Step:
  • <attribute> <is|will be> "<value>"

The next two steps enter this value into the todo field and verify that it was succesfully added to my todo list. Recall that this is the same my todo list again that we bound a selector in the first step definition and confirmed was absent from the page after we navigated to it. We now reuse it here to check that it contains our newly added item (for each record). If it does not an assertion error will be raised.

Step Definition 2, Step 3 and 4
  When I enter my new item in the todo field  Then my todo list should contain my new item
Uses Gwen DSL Steps:
  • I <enter|type> <attribute> in <element>
  • <element|attribute> <should|should not> <be|contain|start with|end with|match regex|match xpath> <attribute>

Passing parameters to step definitions#

The last step definition accepts the number of expected items from the last step in the feature into a parameter placeholder named expected count and then uses that to assert its value against the displayed count.

Feature Step
  Then the list will contain 2 items
Step Definition
  @Assertion  @StepDef  Scenario: the list will contain <expected count> items      Given the displayed count can be located by css ".todo-count strong"       Then the displayed count should be "$<expected count>"

Behavior Rules#

By default, Gwen imposes the following rules to help enforce correct semantics in your Gherkin specs. Violations are reported as errors.

Steps declared with KeywordCan only be bound to StepDefs declared with Behavior Annotation
Given@Context
When@Action
Then@Assertion
Disable at Your Own Discretion

These rules can be disabled in your settings.

Consider the last step in our example:

Last Feature Step
  Then the list will contain 2 items
Step Definition
  @Assertion  @StepDef  Scenario: the list will contain <expected count> items      Given the displayed count can be located by css ".todo-count strong"       Then the displayed count should be "$<expected count>"

This binding is permitted since the feature step is declared with the Then keyword and the step definition is annotated with the compatible @Assertion behavior type.

If we were to change the keyword in the step to When however, it would not only read poorly but would imply action type behavior which would no longer be compabile with @Assertion. The binding would not be permitted and Gwen would report the violation. Similarly, if we were to change the behavior on the step defintion to @Context, it would not be compatile with either When or Then in the step.

3. Launch Gwen to automate#

Launch Gwen to bind your meta and feature specs and automate execution. For the examples presented here, that would be an automated test execution.

Your meta governs what will be evaluated and there are many more things you can do in meta with the Gwen DSL in addition to what is introduced here. Keep in mind also that you need not confine your usage to just testing. You can use Gwen to automate other similarly repetitive and robotic processes too.
