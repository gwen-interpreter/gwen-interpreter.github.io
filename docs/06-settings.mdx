# Settings

> Jump to [Settings Reference](#settings-reference)

## Settings Files

All Gwen settings and properties are maintained in settings files in your project.

### Formats

The following settings file formats are supported:

- [HOCON](https://github.com/lightbend/config/blob/master/HOCON.md#hocon-human-optimized-config-object-notation) (Human-Optimized Config Object Notation), since [v3.0.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v3.0.0)
  - The recommended and default format
  - A superset of JSON
  - Maintained in `.conf` files
- JSON, since [v3.0.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v3.0.0)
  - Pure JSON format
  - Maintained in `.json` files
- Properties, since <Link to="https://github.com/gwen-interpreter/gwen-web/releases/tag/v1.0.0">v1.0.0</Link>
  - Legacy Java properties style format
  - Flat `name = value` pairs
  - Maintained in `.properties` files

Ideally you should use one format for all your settings files, but using combinations of the above is also permitted. Priority will be given in the order shown to same named files in a directory having different formats:

1. `.conf`
2. `.json`
3. `.properties`

### Load order and precedence

Settings are loaded in the following order of precedence, with those higher in the list overriding the lower.

1. [Environment variable](/docs/launcher#environment-variables) overrides
2. [System properties](/docs/launcher#system-properties-and-jvm-options)
3. User settings
    - A `gwen.conf`, `gwen.json`, or `gwen.properties` file in your user directory
4. Profile settings
    - A `*.conf`, `*.json`, or `*.properies` file denoted name passed to [`-p|--profile`](/docs/launcher#gwen-launch-options-profile) launch option.
5. Launch settings
    - Comma separated list of `*.conf`, `*.json`, or `*.properies` files passed to Gwen via the [`-c|--conf`](/docs/launcher#gwen-launch-options-conf) launch option. These are loaded in the order given with latter entries overriding the former.
6. Project settings
    - A `gwen.conf`, `gwen.json` or `gwen.properties` file in your root project directory
7. Default settings
    - Internal Gwen defaults

:::info System properties override all other settings

`-Dname=value` pairs passed to Gwen as system properties have topmost precedence and will always override ALL other settings.

:::

You can maintain settings and override internal defaults at various levels as described here.

### User settings files

User-level settings can optionally be defined in a `gwen.conf|json|properties` file in the root of your user directory on a machine. These are useful when you want to override certain settings on a machine for a user.

```
 /Users
 └── /<username> (~/) 
     └── gwen.conf           # User-level overrides
```

### Profile settings files

All [profile](/docs/profiles) settings are stored in `*.conf|json|properties` files in the `gwen/conf/profiles` subdirectory of the project. Gwen selectively discovers and loads these files when you inovke Gwen with the [`-p|--profile`](/docs/launcher#gwen-launch-options-profile) option passing in a profile name. The settings file with the same name as the passed in profile is the one that is loaded.

```
 /project                     # Your project root
 └── /gwen
     └── /conf
         └── /profiles        # Profile settings
             ├── profile1.conf   
             ├── profile2.conf
             ├── ..
             └── profileN.conf
```

### Project settings files

When you [set up](/docs/get-started) and initialise your project, all project level settings are stored in a `gwen.conf|json|properties` file in the root of the project. Gwen automatically discovers and loads this file for you, so you don't need to explictly pass it to Gwen when you launch in the project directory.

```
 /project                    # Your project root
 └── gwen.conf               # Project settings
```

<details id="gwen-behavior-rules">
  <summary class="setting">

  ```properties
  gwen.behavior.rules
  ```
  <p>
    Controls whether strict or lenient rules around Given-When-Then usage will be enforced for features.
    <br/><br/><em>Note</em>: <code>gwen.behaviour.rules</code> is no longer a supported alias (since v3.0.0)
  </p>
  </summary>
  <p>Supported values</p>
  <ul>
    <li>
      <code>strict</code> to enforce the following strict rules (default):
      <ul>
        <li>Steps in scenarios and backgrounds must satisfy Given-When-Then order</li>
        <li>Givens must set context</li>
        <li>Whens must perform actions</li>
        <li>Thens must assert expectations</li>
        <li>Buts must perform assertions (and are permitted after Then steps)</li>
        <li>
          StepDefs tagged with
          <ul>
            <li><code>@Context</code> : can only be called by Givens</li>
            <li><code>@Action</code> : can only be called by Whens</li>
            <li><code>@Assertion</code> : can only be called by Thens or Buts</li>
          </ul>
        </li>
      </ul>
    </li>
    <li><code>lenient</code> to relax all strict rules</li>
  </ul>
  <Tabs

## Masked Settings

> Since [v2.48.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v2.48.0)

It is a common security practice to never log sensitive data. To support this practice, Gwen will hide all [settings](/docs/settings) that you flag as "masked" by logging their values as `*****` in all logs, reports, error messages and console outputs.

### Flagging masked settings

To flag a setting as masked, append `:masked` to its defined name.

For example, to access a password stored in an environment variable named `USER_PASSOWRD`, you can read it into a masked setting named `user.password` as follows:

<Tabs
    groupId="settings"
    defaultValue="conf"
    values={[
      {label: 'conf', value: 'conf'},
      {label: 'json', value: 'json'},
      {label: 'properties', value: 'properties'}
    ]}>

  <TabItem value="conf">

  ```json
    user {
      "password:masked" = "${env.USER_PASSWORD}"
    }
  ```

  </TabItem>

  <TabItem value="json">

  ```json
    {
      "user": {
        "password:masked": "${env.USER_PASSWORD}"
      }
    }
  ```

  </TabItem>

  <TabItem value="properties">

  ```properties
    user.password\:masked = ${env.USER_PASSWORD}
  ```

  *Note*: You will need to escape `:` as `\:` in properties files

  </TabItem>

  </Tabs>

### Masking input fields

> Since [v4.1.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v4.1.0)

To mask an [input](/docs/data-feeds) field, add the field's name to the [gwen.input.data.maskFields](#gwen-input-data-maskFields) setting. The field value will then be masked in all Gwen outputs.

### Using masked settings

To use a masked setting, just reference it anywhere you need to using it's name (excluding the `:masked` suffix).

Example:

```gherkin
  When I enter "${user.password}" in the password field
```

When evaluated, the above will be logged as follows in all outputs:

```gherkin
  When I enter "*****" in the password field
```

Console output

:::

:::caution

It is your resposibility to enter sensitive data such as passwords into password protected fields to prevent them from being displayed in the browser.

:::

### Custom mask character

You can change the default mask character from `*` to another character of your choice by assigning it to the [`gwen.mask.char`](/docs/settings#gwen-mask-char) setting.

### Masked Data Bindings

Values returned by data bindings in the following DSLs can also be masked with the `@Masked` annotation:

- [`<name> is defined by <javascript|js> "<script>"`](/docs/dsl#name-is-defined-by-javascript-expression)
- [`<name> is defined by <javascriptRef> applied to "<argument>"`](/docs/dsl#name-is-defined-by-js-function-applied-to-arg)
- [`<name> is defined by <javascriptRef> applied to "<arguments>" delimited by "<delimiter>"`](/docs/dsl#name-is-defined-by-js-function-applied-to-args-delimited-by)
- [`<name> is defined by system process "<expression>"`](/docs/dsl#name-is-defined-by-system-process-expression)
- [`<name> is defined by system process "<expression>" delimited by "<delimiters>"`](/docs/dsl#name-is-defined-by-system-process-expression-delimited-by-delimiters)
- [`<name> is defined by unix system process "<expression>"`](/docs/dsl#name-is-defined-by-unix-system-process-expression)
- [`<name> is defined by the <nodeType> in <xmlRef> by xpath "<expression>"`](/docs/dsl#name-is-defined-by-the-node-type-in-xml-ref-by-xpath-expression)
- [`<name> is defined in <textRef> by regex "<expression>"`](/docs/dsl#name-is-defined-in-text-ref-by-regex-expression)
- [`<name> is defined in <jsonRef> by json path "<expression>"`](/docs/dsl#name-is-defined-in-json-ref-by-json-path-expression)
- [`<name> is defined in the <dbName> database by sql "<selectStmt>"`](/docs/dsl#name-is-defined-by-database-select-statement)

## JSONArray Notation

> Since [v3.53.0](https://github.com/gwen-interpreter/gwen-web/releases/tag/v3.53.0)

Lists can be stored in settings in various ways. For example, a list of values could be stored in your settings file as follows:

<Tabs
    groupId="settings"
    defaultValue="conf"
    values={[
      {label: 'conf', value: 'conf'},
      {label: 'json', value: 'json'},
      {label: 'properties', value: 'properties'}
    ]}>

  <TabItem value="conf">

  ```json
    my {
      values = [
        "Gwen"
        "Jenkins"
        "Yarn"
        "Docker"
      ]
    }
  ```

  </TabItem>

  <TabItem value="json">

  ```json
    {
      "my": {
        "values": [ 
          "Gwen", 
          "Jenkins", 
          "Yarn", 
          "Docker"
        ]
      }
    }
  ```

  </TabItem>

  <TabItem value="properties">

  ```properties
    my.values = Gwen,Jenkins,Yarn,Docker
  ```

  </TabItem>

  </Tabs>

To access a list of values configured in your settings as a JSON array, append `:JSONArray` to its setting name in the placeholder references in your feature or [meta](/docs/meta) steps. A JSON array literal containing the values will then be expanded inline.

<p><b>Example 1</b></p>

The following will bind the JSON Array `["Gwen","Jenkins","Yarn","Docker"]` to the `values` variable in the current scope:
```gherkin
  Given values is "${my.values:JSONArray}"
```

Will expand to:

```gherkin
  Given values is "["Gwen","Jenkins","Yarn","Docker"]"
```

<p><b>Example 2</b></p>

The following will inline the JSON Array into a JavaScript expression to get the length and store that in the `length` variable in the current scope:
```gherkin
  Given length is defined by js "${my.values:JSONArray}.length"
   Then length should be "4"
```

Will expand to:
```gherkin
  Given length is defined by js "["Gwen","Jenkins","Yarn","Docker"].length"
   Then length should be "4"
```
